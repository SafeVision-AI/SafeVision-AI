1.1 The Most Important Decision — Separate Service
The chatbot service is built and deployed separately from the main FastAPI backend. This is not optional — it is a hard architectural requirement. Here is exactly why.

📌  WHY: The chatbot loads large AI models into memory — sentence-transformers (500MB), ChromaDB (variable, depends on documents), LangGraph agent runtime, and connections to multiple LLM APIs. If these are loaded inside the main FastAPI backend, every server restart takes 3-5 minutes and the backend becomes too heavy to run on Render.com free tier. A separate chatbot service lets the main backend stay fast (under 30 seconds restart) while the chatbot service can take as long as it needs to initialize.

4. GITHUB BRANCH STRATEGY — HOW 9 PEOPLE BUILD WITHOUT CONFLICTS
With 9 team members touching the same repository, branch management is what prevents your work from destroying someone else's. This section defines the exact branching rules for the chatbot service build. Follow these rules without exception.

4.1 Repository Structure
The project uses a monorepo on GitHub Organization. The chatbot service lives in the chatbot_service/ folder. All chatbot work happens in branches prefixed with chatbot/.


4.2 Chatbot Branch Naming Convention
Every chatbot branch must follow this naming pattern. This prevents confusion and makes it clear what each branch is about.


4.3 Daily Workflow — What Every Team Member Does Every Day
Morning: git pull origin develop to get latest changes from everyone else.
Before starting work: git checkout -b chatbot/your-feature-name to create your branch from current develop.
During work: commit small and often. Commit every time something works, not just when everything works. 'Add Groq provider wrapper with health check' is a good commit message.
When your feature works: git push origin chatbot/your-feature-name — push to GitHub.
Create a Pull Request from your branch to develop. Add a description: what you built, how to test it, what it depends on.
Tag the team lead for review. Lead merges to develop after review.
Never merge your own PR. Always have someone else review. This catches bugs before they reach develop.

4.4 Conflict Prevention Rules
Rule 1: Never edit a file that is not in your assigned ownership area (see Section 1.3 team assignment table)
Rule 2: If you need to change a file owned by someone else, open a conversation in GitHub Issues first
Rule 3: The chatbot_service/ folder is owned by chatbot team — backend team must not touch it
Rule 4: The main backend's api/v1/ folder is owned by backend team — chatbot team calls it via HTTP, does not edit it
Rule 5: Before creating a PR, always rebase your branch on latest develop: git rebase develop
Rule 6: Keep PRs small — one feature per PR. Large PRs have more conflicts and harder reviews

⚠️  The most common conflict scenario: two people both edit chatbot_service/config.py to add their own settings. Prevent this by having one person own config.py (Member A) and everyone else creates an issue asking them to add a new setting. The config owner does it within 24 hours.

6. ALL LLM PROVIDERS — SETUP AND ROUTING
RoadSoS chatbot uses six LLM providers in a priority chain. The router tries them in order, automatically falling back when one is unavailable or rate-limited. This gives the chatbot near-100% uptime at zero cost. All six providers have free tiers.

6.1 Why Six Providers Instead of One
📌  WHY: A single provider has three failure modes: rate limits (too many requests), downtime (rare but happens), and context window limits (long conversations hit token limits). With six providers in a fallback chain, any one of them being unavailable does not affect the user. The router switches to the next provider transparently within 500ms. The user sees a slightly slower response — they never see an error.

6.2 Provider Details — Complete Reference

6.3 The Provider Router — How Fallback Works
The router in chatbot_service/providers/router.py implements a circuit breaker pattern. If a provider returns a rate limit error (HTTP 429) or is down (HTTP 500), the circuit opens for 60 seconds and all requests skip that provider. After 60 seconds, one test request is sent. If it succeeds, the circuit closes and the provider returns to rotation.

Provider Health Check System
Every 60 seconds, a background task in main.py runs a health check against all 6 providers.
Each health check sends a minimal test message: 'Reply with the word OK only.'
If the provider responds within 3 seconds with any response, it is marked healthy in Redis.
The router reads health status from Redis before every request and skips unhealthy providers.
Health status is cached for 60 seconds to avoid hammering rate-limited providers.
The admin endpoint GET /admin/provider-health shows current status of all 6 providers.

6.4 Intent Detection — Separate Cheap Model
Intent classification uses Groq llama3-8b-8192, not the 70B model. The 8B model is 10x cheaper per token and responds in under 200ms. It is accurate enough for classification even though it is not as capable as the 70B for complex reasoning. Every message is classified before the expensive 70B model is called.

📌  WHY: Classifying intent with an 8B model first means 30% of messages can be answered without calling the 70B at all — simple queries like 'hello', 'thank you', or queries where the intent is so clear that pre-fetched tool data is enough. This reduces Groq token usage by roughly 30% and speeds up responses for simple queries.

6.5 NVIDIA NIM — Special Setup
NVIDIA NIM (Neural Interface Models) provides access to optimized large language models through an OpenAI-compatible API. The setup is slightly different from other providers because it uses a different base URL.

Sign up at build.nvidia.com and generate an API key
The base URL is https://integrate.api.nvidia.com/v1 — not api.openai.com
All other parameters (model name format, messages array, max_tokens) follow the OpenAI API format
Use model name: meta/llama-3.1-70b-instruct — note the forward slash, different from Groq's format
Free tier: 1000 API calls. After that, pay-per-use at very low rates. For a hackathon, 1000 calls is more than enough as fallback
NVIDIA also offers: mistralai/mixtral-8x7b-instruct, google/gemma-2-27b-it, and many others — all accessible with the same API key

8. REAL-TIME DATA CONNECTIONS — DETAILED IMPLEMENTATION
This section goes deeper on the three real-time data systems: GPS context injection, WebSocket map push, and the Redis Pub/Sub event pipeline. Each one must be working before the chatbot feels genuinely live.

8.1 GPS Context Injection — Making Every Message Location-Aware
Every chat message from the frontend must include the user's current GPS coordinates. This is how the chatbot knows which hospitals, which police stations, and which roads to talk about. Without GPS, the chatbot can only give generic answers — with GPS, it gives specific actionable responses.

The Context Injection Pipeline
Frontend reads GPS every 30 seconds using navigator.geolocation.watchPosition.
GPS coordinates are stored in Zustand global state.
Every POST /chat/message request includes lat and lon from Zustand state.
Chatbot service receives lat/lon. Before calling any LLM, it calls the emergency tool to get nearby services.
It also calls the road issues tool to check for recent community reports in the area.
All results are formatted into a context string: 'User is in Tambaram, Chennai, TN. Nearest hospital: Apollo (1.2km, 044-28293333, trauma centre). Nearest police: Tambaram Police (3.1km). Recent road issues: 1 pothole reported on NH-44 2 hours ago.'
This context string is prepended to the conversation history in the prompt.
The LLM responds with specific, accurate local information rather than generic advice.

8.2 Redis Pub/Sub — The Real-Time Event Pipeline
When a user submits a road issue report, that information should reach all other users in the area within seconds. The pipeline from report submission to chat context update goes through three systems: PostgreSQL triggers, Redis Pub/Sub, and WebSocket push.

Setting Up PostgreSQL NOTIFY Trigger
In Supabase SQL Editor, run the following logic to set up the trigger. This happens in your Supabase project, not in application code.
Create a trigger function that calls pg_notify with the new road issue data as JSON
The channel name: road_events — must match what Redis subscriber listens to
The payload: JSON containing uuid, issue_type, severity, lat, lon, created_at
Create a TRIGGER on road_issues table AFTER INSERT FOR EACH ROW that calls the trigger function
Test by inserting a row and checking that pg_notify fires — use psql LISTEN road_events to verify

The Redis Subscriber in Chatbot Service
A background asyncio task in the chatbot service main.py subscribes to the Redis channel that receives PostgreSQL notifications. When a new event arrives:
Parse the JSON payload from the notification.
Calculate the GPS grid cell (round lat and lon to 3 decimal places) for the reported location.
Store the event in Redis with key format: road_events:{grid_lat}:{grid_lon} with a 1-hour TTL.
Look up all WebSocket connections that are currently monitoring that grid cell area.
Push the new event to those connections as a JSON message: {type: road_issue, data: {issue details}}.
Frontend receives the WebSocket message and updates the map layer to show the new marker.

8.3 Session Context Refresh
When a new road event arrives for a user's area and that user is actively in a chat session, their next message should automatically have the new event in its context. This happens through the context_assembler.

context_assembler.py checks Redis for road_events:{user_grid_lat}:{user_grid_lon} on every message
If new events are found (created in last 30 minutes), they are included in the context string
The LLM then proactively mentions the new information if relevant: 'Also, a pothole was just reported 800m ahead on your current road'
Events older than 30 minutes are included but marked as older: '2 hours ago: pothole reported near Tambaram'

9. VOICE IMPLEMENTATION — INPUT AND OUTPUT IN DETAIL
Voice is one of the most important accessibility features in RoadSoS. An injured user cannot type. A driver cannot look at a screen. Voice makes the app usable in the most critical scenarios. This section covers exactly how to implement both voice input and voice output.

9.1 Voice Input — Web Speech API Implementation
Browser Compatibility

Voice Input Behaviour Specification
Mic button default state: grey microphone icon. Tap to start recording.
Recording state: red pulsing circle behind the microphone icon. This tells the user the app is listening.
Live transcript: Text appears in the chat input field as the user speaks. This provides real-time feedback.
Auto-stop condition: 5 seconds of silence after the user stops speaking. The timeout is reset every time new speech is detected.
Manual stop: Tapping the mic button again stops recording immediately.
On stop: The transcript is automatically submitted as a chat message — no need to tap Send.
Error handling: If the browser does not support Web Speech API, the mic button is disabled and shows a tooltip 'Use Chrome or Edge for voice input'.
Language detection: Web Speech API accepts a language hint. Default to the user's browser language. If Tamil or Hindi speakers are likely, set recognition language to the most recently detected intent language.

9.2 Voice Output — Web Speech Synthesis
Auto-Read Logic
Not every chatbot response should be read aloud automatically — that would be annoying for normal queries. But emergency responses must be read aloud because the user may not be looking at the screen.

Auto-read trigger conditions: Response contains '112', or 'ambulance', or 'trauma', or 'unconscious', or any first-aid keywords
Auto-read trigger conditions: User's intent was FIND_HOSPITAL, FIND_AMBULANCE, or FIRST_AID_INFO
Auto-read trigger conditions: Crash detection fired in the last 60 seconds
Manual read: Speaker icon button on every bot message — user can tap to hear any response
Stop reading: Tapping anywhere on the screen stops the current speech synthesis
Hands-free mode: When enabled in Settings, ALL responses are auto-read regardless of content

Voice Settings
Rate: 0.9 — slightly slower than default. Important because emergency information needs to be processed clearly.
Pitch: 1.0 — neutral. Not artificially high or low.
Volume: 1.0 — maximum. Emergency information must be audible over road noise.
Language: Matches the detected language of the chatbot's response. Hindi response uses hi-IN voice. Tamil response uses ta-IN voice. English default is en-IN.
Voice selection: On Android, the device's default voice engine provides language-specific voices. The speechSynthesis.getVoices() call returns available voices — select the one matching the response language.

9.3 The Complete Hands-Free Experience
When hands-free mode is on and the user speaks a query, the complete flow is:
User taps the mic button once to activate hands-free session OR hands-free auto-activates when crash detection fires.
User speaks the query. Transcript appears in real time in the input field.
After 5 seconds of silence, transcript is submitted automatically.
Typing indicator shows while the agent processes the query.
Response appears as text in the chat bubble.
Voice synthesis immediately reads the response aloud.
After reading, the mic button automatically activates again — ready for the next question.
User can continue the entire conversation without ever touching the screen.
💡  This hands-free loop is the most technically impressive demo moment after the offline map. Demonstrate it at Stage 2: put the phone face-down on a table, speak 'nearest hospital near Tambaram', and hear the chatbot respond with hospital names and phone numbers. Completely hands-free.

11. DEPLOYMENT — CHATBOT SERVICE ON ITS OWN RENDER INSTANCE
The chatbot service deploys to a separate Render.com web service. This is the most important deployment decision. The main backend and the chatbot service are completely independent deployments. If the chatbot crashes, the emergency map still works. If the main backend updates, the chatbot does not need to redeploy.

🎯  PURPOSE: Independent deployment also means the chatbot can be updated without downtime on the main backend. This is critical for a live app where users rely on the emergency features. Update the legal knowledge base, rebuild ChromaDB, and redeploy the chatbot — the main map keeps working throughout.

11.1 Render.com Setup for Chatbot Service
Creating the Chatbot Service Instance
Log in to render.com. Click New → Web Service.
Connect your GitHub organization repository.
Set Root Directory to chatbot_service/ — this is critical. Without it, Render uses the monorepo root.
Build Command: pip install -r requirements.txt
Start Command: uvicorn main:app --host 0.0.0.0 --port $PORT --workers 1
Instance Type: Free tier (512MB RAM, shared CPU) is sufficient for the chatbot service.
Set all environment variables from Section 2 in the Render environment variables panel.
Set the health check path to /health — Render uses this to determine if the service is running.

Environment Variables in Render Dashboard
Set every variable from Section 2.3 in the Render Environment panel. Important: the MAIN_BACKEND_URL must point to your main backend's Render URL — this is how the chatbot calls the emergency data API.

11.2 ChromaDB in Production — The Critical Decision
ChromaDB stores vector embeddings on disk. In production on Render, disk storage is ephemeral — it resets every time the service restarts. This means you cannot store ChromaDB on the local filesystem in production.

Three Options for Production ChromaDB
💡  For the hackathon, option 1 is correct — build ChromaDB during Docker image build. The Dockerfile runs build_vectorstore.py during the build step. This means the built Docker image already has the vector index. When Render starts the container, ChromaDB loads from the baked-in index in under 5 seconds.

11.3 The Chatbot Service URL
After deployment, the chatbot service is accessible at https://roadsos-chatbot.onrender.com (or whatever name you choose in Render). The frontend NextJS app must be updated to call this URL for chat requests, not the main backend URL.


11.4 GitHub Actions CI/CD for Chatbot Service
The CI/CD pipeline runs tests and deploys automatically when code is merged to develop (staging) or main (production).

Trigger on push to develop: Run all pytest tests in chatbot_service/tests/. If all pass, deploy to staging Render instance.
Trigger on push to main: Run all tests. If all pass, deploy to production Render instance.
Trigger on PR to develop: Run all tests. Post pass/fail status as a PR check. Merging blocked if tests fail.
The workflow file goes in .github/workflows/chatbot-ci.yml — separate from the main backend CI/CD workflow.
